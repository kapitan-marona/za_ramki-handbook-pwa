# RLS migration test plan

Дата: 2026-06-18

Файл миграции: `sql/migrations/2026-06-18_rls_hardening_draft.sql`

Проверки без изменения базы:
`sql/checks/2026-06-18_rls_preflight_checks.sql`

## Что меняет миграция

RLS - это правила доступа к строкам в Supabase. Простыми словами: даже если
человек открыл приложение и знает адрес таблицы, база сама решает, какие строки
ему можно читать, создавать, менять или удалять.

Эта миграция заменяет широкие правила вида "любой авторизованный пользователь
может всё" на правила через реальные роли:

- `admin` - полный доступ к управлению, базе знаний, allowlist и опасным
  действиям;
- `staff` - рабочий доступ к задачам, проектам, комментариям, чек-листам,
  ссылкам и файлам, но без управления исполнителями и доступом задачи;
- пользователь не из `allowlist` - не должен получить рабочие данные.

## Что проверить перед применением

1. Открыть миграцию в Supabase SQL Editor и выполнить сначала через `BEGIN` /
   `ROLLBACK`, если проверка проводится на копии или тестовом проекте.

Перед этим запустить `sql/checks/2026-06-18_rls_preflight_checks.sql`.
Этот файл ничего не меняет, а только показывает риски: неизвестные роли,
задачи `staff` без исполнителей, широкие политики и состояние RLS.

После применения миграции запустить тот же файл ещё раз. В блоке broad policies
для рабочих таблиц не должно остаться правил `true`.

Postflight 2026-06-18:

- broad policies block is empty;
- RLS is enabled on all 17 checked tables;
- all expected functions exist;
- all expected functions are `security definer`;
- all expected functions have `search_path=public`.

2. Проверить вход под админом:
   - открывается Planner;
   - открываются Projects, Articles, Templates, Checklists, Admin;
   - можно создать и обновить задачу;
   - можно создать, обновить и удалить проект;
   - можно создать, обновить и удалить статью, шаблон, чек-лист;
   - открывается список сотрудников/allowlist.

3. Проверить вход под staff:
   - открывается Planner;
   - видны задачи и проекты;
   - видны только задачи с доступом "видно всем" или задачи, где этот
     сотрудник назначен исполнителем;
   - можно сменить статус задачи по текущим правилам;
   - нельзя сменить статус чужой задачи с доступом "только исполнитель" даже
     прямым RPC-вызовом `set_task_status`;
   - можно добавить комментарий;
   - можно удалить свой комментарий;
   - нельзя добавить комментарий к чужой задаче с доступом "только исполнитель"
     прямым RPC-вызовом `add_task_comment`;
   - можно отметить пункт чек-листа только в доступной задаче;
   - можно добавить и удалить ссылку в задаче;
   - видно, кто назначен исполнителем;
   - нельзя назначить исполнителя;
   - нельзя изменить доступ задачи;
   - опубликованные статьи, шаблоны и чек-листы открываются;
   - черновики базы знаний не видны;
   - Admin-раздел недоступен.

4. Проверить вход пользователем, который есть в Supabase Auth, но выключен или
   отсутствует в `allowlist`:
   - приложение не должно показывать рабочие данные;
   - прямые запросы к таблицам не должны возвращать задачи, проекты и базу
     знаний.

5. Проверить push:
   - пользователь может сохранить свою подписку;
   - пользователь не видит чужие push-подписки.

## Намеренно оставленные риски

- `staff` всё ещё может создавать и обновлять проекты. Это сделано, чтобы не
  сломать текущий рабочий процесс вне задач. Если проекты тоже должны стать
  строго админской зоной, нужна отдельная проверка UI-сценариев.
- Прямое создание и прямое редактирование задач в таблице `tasks` ограничено
  админом. Рабочие действия staff должны проходить через RPC-функции с
  внутренними проверками.
- RPC-функции `set_task_status`, `set_task_checklist_done`, `add_task_comment`
  и `log_task_activity` специально переопределяются в RLS-миграции, потому что
  они работают с повышенными правами и иначе могли бы обойти обычные политики
  таблиц.
- `staff` может удалять task links, task files metadata и checklist items. Это
  соответствует текущим действиям интерфейса. Если нужно защищать от случайных
  удалений, лучше заменить hard delete на soft delete отдельной задачей.
- Storage-политики для файлов не входят в эту миграцию. Здесь закрыты только
  строки таблицы `task_files`, а не сами объекты в Supabase Storage.
- Прямой Supabase Functions URL остаётся отдельной задачей в конце плана.

## Как откатывать

Перед применением в продакшне нужно сохранить текущие политики из Supabase.
Если после применения что-то ломается, проще всего временно вернуть старые
политики из `sql/Migration/policies.sql`, а затем точечно ослабить только
сломанный сценарий.
