# Supabase To RF Backend Migration Map

Дата: 2026-06-19

## Зачем это нужно

Этот документ отвечает на вопрос: что именно нужно заменить, если ZA RAMKI
переезжает с Supabase Cloud на российский backend.

Важно: это не новая бизнес-логика. Это карта текущих зависимостей, чтобы перенос
делать по частям и не ломать PWA.

## Главный принцип

Frontend должен продолжать работать через `window.ZRBackend`.

Сегодня `ZRBackend` внутри ходит в Supabase. Позже он может ходить в российский
API. Экраны `planner`, `projects`, `articles`, `templates`, `checklists` не
должны знать, какой сервер используется внутри.

## Рабочие таблицы public

Переносить первыми:

- `profiles` - сотрудники, роли, имена, email.
- `allowlist` - кому разрешён доступ и с какой ролью.
- `tasks` - основные задачи.
- `task_assignees` - исполнители задач.
- `task_activity` - история действий в задаче.
- `task_comments` - комментарии к задачам.
- `task_checklist_items` - пункты чек-листа внутри задачи.
- `checklist_instances` - состояние чек-листа внутри задачи.
- `task_links` - ссылки, прикреплённые к задаче.
- `task_files` - файлы, прикреплённые к задаче.
- `projects` - проекты.
- `project_comments` - комментарии к проектам.
- `project_links` - ссылки проекта.
- `kb_articles` - инструкции.
- `kb_templates` - шаблоны.
- `kb_checklists` - чек-листы.
- `push_subscriptions` - подписки браузеров на push.

Переносить позже или проверять отдельно:

- `checklist_templates`
- `checklist_template_items`
- `task_checklists`
- `public_requests`

Не переносить в первую очередь:

- `backup_*` таблицы - это старые резервные копии, они не должны управлять
  текущим интерфейсом.
- системные схемы Supabase: `auth`, `realtime`, `storage`, `vault`,
  `extensions`. Для российского backend у них будут другие аналоги.

## RPC-функции, которые сейчас важны

Эти функции сейчас заменяют прямое изменение некоторых таблиц. При переносе на
свой backend они станут обычными серверными endpoint или внутренними сервисными
методами.

Переносить первыми:

- `get_role` - узнать роль текущего пользователя.
- `is_admin` - проверить, админ ли пользователь.
- `is_staff` - проверить, сотрудник ли пользователь.
- `create_task_admin` - создать задачу админом.
- `update_task_admin` - изменить задачу админом.
- `set_task_status` - изменить статус задачи.
- `set_task_assignee` - назначить исполнителя.
- `set_task_primary_assignee` - назначить основного исполнителя.
- `log_task_activity` - записать действие в историю.
- `add_task_comment` - добавить комментарий к задаче.
- `set_task_checklist_done` - отметить пункт чек-листа.
- `archive_task` - архивировать одну задачу.
- `archive_done_tasks` - архивировать завершённые задачи.
- `can_read_task` - проверить, видна ли задача пользователю.

Переносить позже:

- `create_task_checklist_from_template`
- `auth_email`
- `handle_new_user`
- `set_updated_at`
- `touch_updated_at`

## Модули приложения и что им нужно

### Auth

Нужно заменить:

- вход по email/password;
- хранение сессии;
- обновление сессии;
- выход;
- получение роли.

Файлы frontend:

- `js/utils/supabase_client.js`
- `js/services/zr_backend_provider.js`
- `js/services/zr_backend.js`
- `js/app.js`
- `js/views/login.js`

### Planner

Нужно заменить:

- загрузку списка задач;
- загрузку одной задачи;
- создание и изменение задачи;
- статусы;
- исполнителей;
- комментарии;
- ссылки;
- файлы;
- историю действий;
- чек-листы внутри задачи.

Таблицы:

- `tasks`
- `task_assignees`
- `task_activity`
- `task_comments`
- `task_checklist_items`
- `checklist_instances`
- `task_links`
- `task_files`

### Projects

Нужно заменить:

- список проектов;
- карточку проекта;
- задачи проекта;
- ссылки проекта;
- комментарии проекта.

Таблицы:

- `projects`
- `project_links`
- `project_comments`
- `tasks`

### Knowledge Base

Нужно заменить:

- список инструкций;
- содержимое инструкции;
- список шаблонов;
- список чек-листов;
- админское создание, редактирование и удаление.

Таблицы:

- `kb_articles`
- `kb_templates`
- `kb_checklists`

### Admin

Нужно заменить:

- сотрудников;
- allowlist;
- управление проектами;
- управление инструкциями, шаблонами и чек-листами.

Таблицы:

- `profiles`
- `allowlist`
- `projects`
- `kb_articles`
- `kb_templates`
- `kb_checklists`

### Storage

Supabase Storage нужно заменить на S3-compatible storage или файловый backend.

Что важно сохранить:

- публичные URL для картинок в инструкциях;
- загрузку файлов к задачам;
- права доступа к файлам;
- понятный путь файла, чтобы PWA могла показывать его без лишних переходов.

### Push

Сейчас push ещё зависит от Supabase Functions. Это намеренно оставлено на конец
списка.

Нужен будущий endpoint:

- `POST /push/send`
- `POST /push/subscriptions`
- `DELETE /push/subscriptions/:endpoint`

## Порядок переноса без резкого рывка

1. Оставить текущий Supabase backend рабочим.
2. Подготовить `ZRBackendProvider` как точку переключения.
3. Сделать российский backend-прототип с `GET /health` и `GET /me`.
4. Подключить только чтение задач через новый backend в тестовом режиме.
5. Затем добавить создание/изменение задач.
6. Затем перенести проекты.
7. Затем перенести базу знаний.
8. Затем storage.
9. Затем push и direct Supabase Functions URL.

## Риски

- Auth нельзя переносить “наполовину” без плана: если сессия ломается,
  пользователь не войдёт в приложение.
- Права staff/admin должны повторять текущие RLS-правила. Иначе staff может
  увидеть или изменить лишнее.
- Storage нужно тестировать отдельно: PWA может открываться, но картинки и файлы
  будут грузиться плохо, если URL недоступны из РФ.
- Push лучше переносить последним, потому что основная работа системы не должна
  зависеть от уведомлений.
